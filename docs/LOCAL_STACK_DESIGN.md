# Локальный стек (ПК): проекты, UI, соединение с сервером

Этот документ **проектирует** компоненты, которых ещё нет в полном виде в коде: постоянное соединение рабочей станции с хостом, мультипроектность и отдельный веб-интерфейс на ПК. Реализация может идти поэтапно поверх текущего [`DeployService`](../proto/deploy.proto).

## Цели

1. **Соединение** между машиной с установленным серверным стеком и локальным ПК (NAT, доверие, шифрование) — без смешивания с операторским CLI [`local-stack/client`](../local-stack/client/) (см. [`LOCAL_GATEWAY.md`](LOCAL_GATEWAY.md)).
2. **Локальный UI** для создания «проектов», загрузки собранного архива, указания версии и запуска — по смыслу ближе к продуктовому приложению на рабочей станции, а не к админ-дашборду nginx на сервере.
3. **Модель данных «проекты»** — сейчас сервер оперирует одним `DEPLOY_ROOT` и одним потоком релизов; для нескольких проектов на одном хосте потребуются либо **несколько корней/имён** на сервере (новые таблицы и API), либо соглашение «один сервер — один проект» и **несколько целевых серверов** в локальном UI.

## Реализовано: `pirate-client` (desktop UI)

Локальный бинарь **`pirate-client`** (Tauri 2 + crate `pirate-desktop`) встраивает WebView со статикой из [`local-stack/desktop-ui`](../local-stack/desktop-ui/); статус и данные для UI — через `invoke`, без отдельного HTTP-сервера для SPA. Подробности: [`DESKTOP_CLIENT.md`](DESKTOP_CLIENT.md).

## Предлагаемые компоненты

### `local-agent` (long-lived)

- Отдельный процесс на ПК: исходящее соединение (или VPN/SSH — как в LOCAL_GATEWAY), опционально mTLS.
- Не заменяет разовый вызов `client deploy`; может проксировать HTTP/gRPC к `control-api` / `deploy-server` только после аутентификации.
- Текущий crate [`local-stack/local-agent`](../local-stack/local-agent/) — **заглушка** до появления протокола и политики безопасности.

### `frontend-local` (отдельное приложение)

- Отдельный Vite/React/Vue проект **или** второй entrypoint рядом с серверным UI в `server-stack/frontend/` с отдельной сборкой; целевой каталог — например `local-stack/frontend-local/` (env `VITE_*`).
- Базовый URL API: не относительный `/api` как у серверного дашборда за nginx, а явный `https://<server>/...` или `http://127.0.0.1:<локальный прокси агента>`.
- CORS: серверный `control-api` настраивается через `CONTROL_API_CORS_ORIGINS` / dev-режим (см. PHASE6).

### Модель «проектов»

| Вариант | Плюсы | Минусы |
|--------|--------|--------|
| A. Один проект на сервер | Минимум изменений в БД и API | Несколько «проектов» = несколько серверов в UI |
| B. Несколько имён проектов → отдельные поддеревья `/deploy/<project>/...` или строка в БД | Один хост — много приложений | Расширение proto/API, миграции, права |

Рекомендация: начать с **варианта A** + список «удалённых серверов» в локальном UI; перейти к B при необходимости.

## Зависимости от сервера

- Загрузка артефакта и версии остаётся на **gRPC `DeployService`** до появления отдельного upload через агента.
- История и nginx — зона **серверного** [`control-api`](../server-stack/control-api/); локальный UI либо вызывает их по сети, либо только через агента.

# ROADMAP: Artifact-based deployment (gRPC, Rust)

Минимально жизнеспособная система деплоя: локальный CLI собирает/упаковывает артефакт, сервер принимает поток, хранит версии, разворачивает и управляет процессом. Взаимодействие **только по gRPC**; сеть **только IPv6**.

---

## Цель и критерий успеха

- `client deploy ./build` → сервер принимает архив, проверяет хеш, распаковывает, переключает `current`, запускает приложение.
- `client status` → актуальная версия и состояние процесса (`running` / `stopped` / `error`).
- `client rollback <version>` → откат на указанную сохранённую версию с перезапуском.
- Без Docker на первом этапе, без БД, без лишней архитектуры.

---

## Технологический стек (зафиксировано)

| Компонент | Выбор |
|-----------|--------|
| Язык | Rust |
| gRPC | `tonic` + `prost` |
| Async | `tokio` |
| CLI | `clap` |
| Архив | `tar` + `flate2` (gzip) |
| Хеш | SHA-256 (`sha2`) |
| Логи | `tracing` + `tracing-subscriber` |

---

## Сетевые требования (критично)

- Сервер: bind **`[::]:50051`** (явный IPv6 dual-stack socket; на большинстве ОС `[::]` слушает и IPv4-mapped, но **клиент** подключается только к IPv6-литералу вида `[addr]:port` — без IPv4 fallback в коде клиента).
- Клиент: endpoint в формате **`http://[ipv6]:50051`** (или эквивалент для `tonic` transport).
- В README/скриптах тестов: явная проверка, что используется IPv6 (например `::1` для loopback).

## Файловая схема на сервере

```
/deploy
  releases/
    v1/
    v2/
    ...
  current -> symlink на releases/vN
```

Артефакт: поток байтов → временный файл → проверка SHA-256 (хеш передаётся в контракте или вычисляется на сервере — уточнение на этапе proto: минимум — хеш в финальном сообщении или отдельное поле в `DeployChunk` / метаданные до `is_last`).

---

## gRPC контракт (целевое состояние)

**Сервис `DeployService`:**

1. **`Upload(stream DeployChunk) → DeployResponse`**
   - `DeployChunk`: `bytes data`, `string version`, `bool is_last`
   - `DeployResponse`: `string status`, `string deployed_version`
   - (При необходимости расширение proto: поле `sha256` в последнем чанке или отдельное сообщение метаданных — без усложнения схемы.)

2. **`GetStatus(StatusRequest) → StatusResponse`**
   - `StatusResponse`: `current_version`, `state` (`running` / `stopped` / `error`)

3. **`Rollback(RollbackRequest) → RollbackResponse`**
   - `RollbackRequest`: `version`

Файл: `proto/deploy.proto` → генерация через `tonic-build` в `build.rs`.

---

## Логика деплоя на сервере (порядок операций)

1. Принять stream, записать во временный файл (с лимитом суммарного размера).
2. Проверить SHA-256 (после получения всех данных / последнего чанка).
3. Распаковать в новую директорию `releases/<version>/`.
4. Остановить текущий управляемый процесс (если был запущен).
5. Обновить symlink `current` → `releases/<version>`.
6. Запустить новую версию в фоне (не блокировать gRPC worker).

**Запуск приложения:**

- Предпочтительно: `run.sh` в корне развёрнутого релиза (или в `current`).
- Fallback: исполняемый бинарник с известным именем (конфигурируемо константой/ENV, минимально).

Процесс: отдельная задача `tokio` / `std::process::Child` с возможностью `kill` и рестарта при rollback.

---

## Безопасность (минимум)

- **Лимит размера** upload (константа + возможно ENV).
- **Валидация `version`**: только безопасные символы (например `[a-zA-Z0-9._-]+`), запрет `..`, `/`, `\`, пустой строки.
- **Не root**: при старте сервера проверка UID ≠ 0; при запуске дочернего процесса — от имени текущего непривилегированного пользователя (без `setuid`).

## Этапы реализации (пошагово)

### Этап 0 — ROADMAP (текущий документ)

- Зафиксировать требования, порядок работ, критерии успеха.

### Этап 1 — Proto

- [ ] Написать `proto/deploy.proto` (все RPC и сообщения из ТЗ).
- [ ] Подключить `tonic-build` в `build.rs`, workspace `Cargo.toml`.
- [ ] Сгенерировать код, убедиться что компилируется пустой `lib` или заглушки.

**Следующий шаг после этапа 1:** скелет сервера с `tonic::transport::Server` на `[::]:50051` и заглушками сервиса.

### Этап 2 — Сервер (скелет + сеть)

- [ ] `main.rs`: инициализация `tracing`, парсинг аргументов (порт, корень `/deploy`, лимиты).
- [ ] Явный bind на IPv6 (`[::]:PORT`).
- [ ] Реализовать заглушки `Upload`, `GetStatus`, `Rollback` с логированием входа.

**Следующий шаг:** полная логика деплоя в `deploy_service.rs`.

### Этап 3 — Деплой-логика на сервере

- [ ] Приём stream → temp file + лимит размера.
- [ ] SHA-256 проверка (уточнить поле в proto при реализации: хеш от клиента в последнем чанке или отдельное поле).
- [ ] Распаковка tar.gz в `releases/<version>/`.
- [ ] Управление процессом: старт `run.sh` или бинарника, хранение `Child`, kill при новом деплое/rollback.
- [ ] Symlink `current`, атомарная смена где возможно.
- [ ] Валидация версии, проверка non-root.

**Следующий шаг:** CLI клиент.

### Этап 4 — CLI клиент

- [ ] `clap`: подкоманды `deploy`, `status`, `rollback`.
- [ ] `deploy`: tar.gz каталога `build/`, chunked upload, IPv6 endpoint.
- [ ] `status` / `rollback`: unary вызовы.

**Следующий шаг:** интеграционные сценарии и документация запуска.

### Этап 5 — Тестирование и сценарии

- [ ] Скрипт или инструкция: локальный запуск сервера (не Docker).
- [ ] Тестовый проект: простой `run.sh` + файлы в `build/`.
- [ ] Сценарий: deploy → status → rollback → status.

## Этап 6+ (расширение): установка «под ключ» и веб-интерфейс

Цель: при **установке клиента/пакета для сервера** (или отдельной команды установки, например `install` / `bootstrap`) поднимается не только `deploy-server`, а согласованный стек, чтобы оператор мог управлять деплоем и **наблюдать состояние** через браузер.

### Компоненты стека

| Компонент | Роль |
|-----------|------|
| **deploy-server** (существующий) | gRPC: приём артефактов, релизы, `current`, запуск приложения. |
| **PostgreSQL** | Учёт метаданных: зарегистрированные сервисы/релизы, версии, время деплоя, опционально история событий. Персистентность между рестартами UI и сервера. |
| **Nginx** | Обратный прокси: статика фронтенда, маршрут к HTTP API (если появится поверх gRPC), TLS (опционально), единая точка входа по IPv6. |
| **Frontend** | Веб-UI: список сервисов (или целевого деплоя), **какие процессы/релизы активны**, **версии**, состояние (`running` / `stopped` / `error`), ссылки на детали. |

### Поведение при установке

1. Инициализация/миграции схемы БД (PostgreSQL).
2. Запуск или подключение к уже поднятому `deploy-server` (тот же хост или отдельный сервис — уточняется в дизайне).
3. Старт **nginx** с конфигом: раздача фронтенда + проксирование на backend HTTP (если добавлен) или gRPC-web/REST-адаптер.
4. Сборка/выкладка **frontend** (статика в каталог, который отдаёт nginx).

Конкретика упаковки (systemd, скрипты, пакетный менеджер) — на этапе проектирования установщика; **Docker на первом этапе MVP по-прежнему не обязателен**, но для этапа 6+ допустимы контейнеры или bare-metal — фиксируется отдельным решением.

### Данные для UI (версии и «что запущено»)

- **Минимум:** фронтенд опрашивает backend, который агрегирует ответы существующего `GetStatus` (и при расширении proto — список релизов из ФС `releases/` + состояние процесса).
- **С PostgreSQL:** версии и события деплоя дублируются/логируются в БД для быстрых списков и фильтров без сканирования диска; источником истины для «что сейчас в `current`» остаётся файловая схема + процесс, БД — для отображения и аудита.

### Сеть

- Публичный/операторский вход: **nginx** на выбранном порту (например `[::]:443` или `[::]:80`), внутренний gRPC deploy-server — по-прежнему IPv6 (loopback или приватная сеть).

### Критерий успеха (этап 6+)

- После установки открывается веб-интерфейс и видно **активную версию** и **состояние** деплоя (как минимум то же, что `client status`).
- В списке отображаются **сервисы/релизы** (как минимум один целевой сервис — развёрнутое приложение) и их **версии** согласованно с сервером деплоя.

## Запрещено / ограничения по фазам

**MVP (этапы 0–5):**

- Docker не обязателен и не является целевой платформой первой поставки.
- Отдельная БД **не входит** в минимальный контур (см. этап 6+ для PostgreSQL).
- Излишнее усложнение без запроса (очереди, оркестраторы, отдельный агент).

**Этап 6+:** допускаются nginx, PostgreSQL, фронтенд и иной способ упаковки/развёртывания — по отдельному дизайну и критериям приёмки.

---

## Риски и решения

| Риск | Митigation |
|------|------------|
| Хеш не в первой версии proto | Добавить поле `artifact_sha256` в `DeployChunk` (только в последнем чанке) или в `DeployResponse` только проверка на сервере — минимально: клиент шлёт хеш в последнем чанке. |
| `[::]` и IPv4 | Клиент не использует IPv4 литералы; документация только IPv6. |
| Большие файлы | Streaming + лимит + temp на диске. |
